# 世界杯预测模型：开源算法与许可证调研

创建日期：2026-06-13

## 这份文档的目的

这份文档用于记录世界杯动态预测系统底层算法的可用性判断，重点回答三个问题：

1. 蒙特卡洛模拟、Dixon-Coles、Poisson 这些方法能不能用于客户项目。
2. 现成开源项目和建模库能不能直接使用。
3. 如果要做成付费预测页面，哪些地方需要自己实现，哪些地方可以参考或引用。

结论先行：**这些方法可以用，而且适合做底层概率引擎；但不能直接复制别人的页面、数据、商业结果或无授权代码。**

## 一、总体结论

建议把底层预测系统定义为：

```text
Elo / 实力评分
→ Poisson 或 Dixon-Coles 单场比分模型
→ 蒙特卡洛模拟整届世界杯
→ 叠加赛程路径、伤病、新闻、教练、关键球员、旅行等修正因子
→ 用五大盘面解释给用户
```

其中：

- **蒙特卡洛模拟**是公开数学方法，不是某个软件，自己实现没有授权问题。
- **Poisson / Dixon-Coles**是公开统计建模方法，可以按照公式自己实现。
- **MIT License 的项目或库**可以作为参考或依赖，但要保留版权和许可证说明。
- **GPL-3.0 项目**不建议直接引入闭源商用项目，除非能接受对应开源义务。
- **无 License 的 GitHub 项目**不要直接复制或改造代码。
- **Football Meets Data 这类商业平台**只能参考方法和产品表达，不能直接复制数据、页面、预测结果或模拟器代码。

## 二、蒙特卡洛模拟是否能用

### 结论

可以用，而且应该用。

世界杯不是普通联赛，最终概率高度依赖晋级路径。2026 年世界杯有 48 支球队、12 个小组、32 强淘汰赛、小组第三出线机制，路径组合非常复杂。只看某支球队的基础实力，无法解释它实际夺冠概率。

蒙特卡洛模拟适合解决这个问题：把整届赛事随机跑很多次，每一次都根据单场胜平负概率模拟结果，最后统计每支球队的出线、晋级、夺冠概率。

### 在本项目里的作用

蒙特卡洛模拟应该承担这些任务：

- 模拟小组赛最终排名。
- 处理小组第三出线组合。
- 模拟 32 强、16 强、8 强、4 强、决赛。
- 统计每队进入每一轮的概率。
- 在比赛结束后锁定真实结果，只模拟剩余比赛。
- 在球队被淘汰后把对应概率归零。
- 在赛程路径变化后重新计算所有球队概率。

### 建议模拟次数

不同阶段可以使用不同模拟次数：

| 阶段 | 建议模拟次数 | 说明 |
|---|---:|---|
| 开发调试 | 5,000 - 10,000 次 | 速度快，方便验证逻辑 |
| 正式每日更新 | 50,000 次 | 精度和速度相对平衡 |
| 重要节点更新 | 100,000 次以上 | 比如小组赛结束、淘汰赛开始前 |

### 注意事项

- 模拟次数越多，结果越稳定，但计算时间更长。
- 不能只模拟冠军概率，也要模拟每轮晋级概率。
- 每次真实比赛结束后，不能重新随机已经结束的比赛，必须锁定赛果。
- 小组第三出线机制要认真处理，否则路径概率会错。

## 三、Poisson 与 Dixon-Coles 是否能用

### Poisson 模型

Poisson 模型用于估算一场比赛双方可能进几个球。它的基本思路是：

```text
球队 A 预期进球 = A 的进攻强度 × B 的防守弱点 × 场地/主客/中立修正
球队 B 预期进球 = B 的进攻强度 × A 的防守弱点 × 场地/主客/中立修正
```

得到双方预期进球后，可以计算 0-0、1-0、1-1、2-1 等比分概率，再汇总成胜平负概率。

优点：

- 实现简单。
- 易解释。
- 适合和 Elo、xG、攻防评分结合。

缺点：

- 默认两队进球独立。
- 对足球低比分相关性处理不够细。
- 对 0-0、1-0、0-1、1-1 这类低比分场景可能不够准确。

### Dixon-Coles 模型

Dixon-Coles 是 Dixon 和 Coles 在 1997 年论文《Modelling Association Football Scores and Inefficiencies in the Football Betting Market》中提出的足球比分模型。它本质上是在 Poisson 模型基础上，对低比分结果做相关性修正。

它适合足球，因为足球经常出现低比分，而低比分之间不是完全独立随机的。比如 0-0、1-0、0-1、1-1 这类比分在实际比赛里有特殊相关性。

优点：

- 比普通 Poisson 更适合低比分足球比赛。
- 可以估算胜平负和具体比分概率。
- 适合放进蒙特卡洛模拟中作为单场比赛概率来源。

缺点：

- 对数据质量要求更高。
- 国家队比赛样本较少，容易受近期比赛稀疏影响。
- 如果没有足够比赛数据，参数估计会不稳定。

### 本项目建议

建议采用保守方案：

```text
Elo / 实力评分负责基础强弱
Poisson 或 Dixon-Coles 负责把强弱差转成单场比分概率
蒙特卡洛负责整届赛事路径模拟
新闻与人工因子负责赛前修正和解释
```

不要一开始就把伤病、士气、教练、更衣室、天气全部硬塞进 Dixon-Coles 公式。更稳妥的方式是先算基础胜率，再做小幅修正。

示例：

```text
基础单场胜率：法国 62%
核心前锋缺阵：-4%
连续远距离旅行：-1%
对手主力中卫停赛：+2%
最终胜率：59%
```

## 四、现成项目和库的可用性

### 1. Cup26 AI / world-cup-2026-prediction-model

地址：
https://github.com/Hicruben/world-cup-2026-prediction-model

公开信息：

- GitHub 显示为 MIT License。
- README 明确写的是 `Elo ratings → Dixon-Coles bivariate Poisson → Monte Carlo simulation`。
- 项目说明中提到 50,000 次模拟。
- 项目会锁定已结束比赛。
- 淘汰球队概率会归零。
- 会处理 2026 年小组第三出线路径。
- 项目还提供 backtest，并使用 RPS、log-loss、Brier score、ECE 等指标评估模型。

可用性判断：

| 用法 | 是否建议 | 说明 |
|---|---|---|
| 学习整体结构 | 建议 | 它的底层结构和本项目方向很接近 |
| 参考 MIT 代码 | 可以 | 需要保留版权和许可证说明 |
| 直接复制页面设计 | 不建议 | 页面和品牌表达应该自己做 |
| 直接复制预测结果 | 不建议 | 客户项目应该有自己的模型输出 |
| 作为底层依赖 | 可以，但要谨慎 | 要确认代码质量、更新机制、数据来源 |

对本项目的价值：

- 可以参考它如何把 Elo、Dixon-Coles、蒙特卡洛串起来。
- 可以参考它如何锁定已完成比赛。
- 可以参考它如何处理小组第三出线。
- 可以参考它如何做 backtest。

风险：

- 如果直接依赖外部项目，后续维护不受我们控制。
- 如果直接搬结果，产品会缺少独立性。
- 需要检查它的数据来源是否适合商用。

建议：

**把它作为结构参考，不要把它作为客户项目的品牌核心。**

### 2. penaltyblog

地址：

公开信息：

- GitHub 显示为 MIT License。
- Python 足球分析库。
- 包含 Poisson、Bivariate Poisson、Dixon-Coles 等模型。
- 面向足球数据分析、比赛建模和预测。

可用性判断：

| 用法 | 是否建议 | 说明 |
|---|---|---|
| 作为 Python 建模库 | 建议评估 | 比单个世界杯项目更通用 |
| 学习模型接口 | 建议 | 可参考它如何组织 football modelling |
| 直接商用依赖 | 可以 | MIT 通常允许商用，但要保留许可证 |
| 复制部分代码 | 可以，但要合规 | 需要保留版权和 MIT License |

对本项目的价值：

- 适合作为后端模型层参考。
- 如果客户项目用 Python/FastAPI，penaltyblog 的技术栈更匹配。
- 可以减少从零实现 Poisson / Dixon-Coles 的时间。

风险：

- 需要确认依赖包大小、部署复杂度和运行速度。
- 需要确认是否适合国家队数据，而不只是俱乐部联赛数据。
- 模型输入数据仍然要自己准备，库本身不能解决数据授权。

建议：

如果要快速做出可运行后端，可以评估使用 penaltyblog。
如果客户预算足够，也可以自己实现简化版 Poisson / Dixon-Coles，降低依赖风险。

## 五、许可证判断

### MIT License

MIT 是宽松开源许可证，通常允许：

- 商用。
- 修改。
- 分发。
- 私有使用。
- 闭源使用。

主要义务：

- 保留原作者版权声明。
- 保留许可证文本。

适合本项目的用法：

- 使用 MIT 库作为依赖。
- 参考 MIT 项目代码结构。
- 复制少量 MIT 代码并保留许可证说明。

仍需注意：

- MIT 解决的是代码许可证，不代表数据也能商用。
- 如果项目里混有第三方数据，要单独看数据授权。

### GPL-3.0

GPL-3.0 是强 copyleft 许可证。它允许商用，但如果分发基于 GPL 代码的作品，通常需要开放对应源代码，并使用同样许可证。

对本项目的影响：

- 如果客户项目要闭源，不建议直接引入 GPL-3.0 代码。
- 如果只是看它的思路，不复制代码，风险较低。
- 如果用 GPL 代码做线上服务，边界要具体看使用方式和分发方式，但商业项目最好避免。

### 无 License 项目

GitHub 上公开不等于可以自由使用。没有许可证时，默认版权仍属于作者。

处理原则：

- 可以阅读学习。
- 不要复制代码。
- 不要改造后商用。
- 如需使用，联系作者授权。

## 六、算法与数据要分开看

这是本项目最容易踩坑的地方。

算法可以自己实现，不等于数据可以随便用。

### 算法层

通常可以自己实现：

- Elo。
- Poisson。
- Dixon-Coles。
- 蒙特卡洛。
- 路径模拟。
- 简单伤病修正。
- 新闻来源评级。

### 数据层

需要单独确认授权：

- 赛程。
- 比分。
- 积分榜。
- 球队名单。
- 球员伤病。
- 红黄牌。
- 首发名单。
- xG。
- Opta / StatsBomb 数据。
- 赔率。
- 预测市场价格。
- 新闻全文。

### 相对安全的数据

这些通常更容易使用，但仍建议确认来源条款：

- 官方赛程和比赛结果。
- 球队名称。
- 已公开比分。
- 已公开积分。
- 官方红黄牌和停赛。
- 官方发布会摘要。

### 高风险数据

这些不能随便抓来商用：

- Opta 详细事件数据。
- StatsBomb 付费数据。
- 商业赔率 API。
- 付费媒体全文。
- Football Meets Data 的模拟结果。
- 第三方平台的专有概率。

建议：

产品第一版可以先用公开赛程、比分、球队基础评分、自建概率模型和手动新闻事件录入。等客户付费和流量跑起来，再考虑购买正式体育数据 API。

## 七、本项目推荐技术方案

### 最小可行版本

第一版不需要过度复杂，目标是能跑、能解释、能每日更新。

```text
输入：
- 48 支球队基础 Elo / 实力评分
- 赛程表
- 已结束比赛结果
- 手动录入或半自动抓取的伤病/停赛/新闻事件

模型：
- Elo 转单场胜平负概率
- Poisson 或简化 Dixon-Coles 转比分概率
- 蒙特卡洛模拟 50,000 次
- 新闻事件做小幅修正

输出：
- 夺冠概率
- 晋级概率
- 每队五大盘面解释
- 每日概率变化原因
```

### 进阶版本

当基础产品跑通后，再加入：

- xG 数据。
- 球员级别状态数据。
- 首发预测。
- 赛前 1 小时官方首发修正。
- 自动新闻抓取和事件抽取。
- 新闻来源可信度评分。
- 预测模型回测。
- 概率变化可视化。

### 不建议第一版就做的事

- 不要一开始就做复杂机器学习黑箱模型。
- 不要把所有新闻传闻都直接喂进公式。
- 不要依赖无授权商业数据。
- 不要直接复制 Football Meets Data 或 Cup26 AI 的页面。
- 不要直接接 GPL-3.0 代码进闭源项目。

## 八、给客户的表达方式

可以这样对客户解释：

> 我们底层会用公开、成熟的足球预测数学方法：球队实力评分、单场比分概率模型和蒙特卡洛模拟。它们不是某个闭源软件，而是可以自己实现的统计方法。然后我们会在这个稳定底层上叠加世界杯特有的路径、伤病、停赛、新闻、旅程和关键球员因素，让页面每天根据真实比赛进展动态更新。

也可以更产品化地说：

> 不是简单写一篇预测文章，而是做一个会随比赛推进自动变化的冠军概率系统。底层负责算概率，上层负责解释为什么变。

需要避免的说法：

- 不要说“我们直接接 Opta 数据”，除非真的购买授权。
- 不要说“我们复制 Football Meets Data 的模型”，应该说参考行业常见概率模拟思路。
- 不要说“AI 掌握内幕”，只能说基于公开信息和权威新闻动态更新。
- 不要说“保证预测准确”，只能说提供概率分析和风险提示。

## 九、最终建议

这些调研结果可以用于本项目，而且非常适合做底层模型。

最终建议路线：

```text
自己实现核心逻辑：
Elo / 实力评分
→ Dixon-Coles 或 Poisson 单场比分概率
→ 蒙特卡洛模拟整届世界杯
→ 叠加伤病、教练、路径、旅行、关键球员、新闻事件修正
→ 前端用五大盘面解释
```

开源使用建议：

- Cup26 AI：参考结构，可参考 MIT 代码，但不要直接复制产品结果和页面。
- penaltyblog：可以评估作为 Python 建模库，适合严肃建模。
- regista：只看思路，不直接引入闭源客户项目。
- Football Meets Data：只做产品和方法论参考，不复制数据和模拟器。

最重要的边界：

**算法方法可以用，代码看 License，数据看授权。**
