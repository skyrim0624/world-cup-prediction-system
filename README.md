# 2026 世界杯动态预测系统

这是一个面向客户验收和后续 Agent 接手的世界杯动态预测产品原型。它不是静态文章，而是把单场赛前预测、比分概率、路径传导、冠军概率变化、新闻事件影响和付费解锁流程放在同一个响应式 Web App 里。

线上验收地址：[world-cup-prediction-system.pages.dev](https://world-cup-prediction-system.pages.dev/)

GitHub 仓库：[skyrim0624/world-cup-prediction-system](https://github.com/skyrim0624/world-cup-prediction-system)

## 当前状态

- 前端：React + TypeScript + Vite，已部署到 Cloudflare Pages。
- 后端：Python + FastAPI，已有 Poisson 单场比分模型、蒙特卡洛路径模拟、事件权重、后台录入和付费订单接口。
- 数据：当前是 2026 赛制骨架 + 样例球队/赛程/事件，可替换为真实授权数据。
- 线上版本：目前 Cloudflare Pages 是静态前端验收版，生产 API、日更任务、支付回调和用户权限仍需下一阶段部署。
- 产品边界：这是概率分析产品，不是投注建议产品；不承诺命中，不使用未授权商业数据，不暗示内幕信息。

## 预测方法

本系统不是直接让 AI 猜比分，而是用一条可复算的概率链路生成预测，再让前端把结果翻译成客户和博主能看懂的表达。

整体链路如下：

```text
球队基础评分 + 赛程 + 已结束赛果
→ 新闻 / 事件 / 赛前上下文修正
→ Poisson 单场比分分布
→ 胜 / 平 / 负、比分矩阵、进球市场视角
→ 蒙特卡洛整届赛事模拟
→ 晋级概率、冠军概率、路径传导和每日变化原因
```

### 1. 三层模型结构

第一层是底层概率模型，负责给出稳定数字。

- 用 `elo` 表示球队基础强弱。
- 用 `attack`、`defense`、`goalkeeper`、`squad` 表示攻防、门将和阵容可用性。
- 用 `path` 表示赛程路径和半区难度。
- 用 Poisson 模型把双方强弱差转成每个比分的概率。
- 用蒙特卡洛模拟把单场概率传导到整届赛事，输出 32 强、16 强、8 强、4 强、决赛和冠军概率。

第二层是事件与新闻处理层，负责处理动态变化。

- 比赛已经结束后，真实比分会锁定进积分表，不再预测这场比赛。
- 进行中比赛会把当前比分作为动态权重，再保守采样剩余进球。
- 新闻会先被转成结构化事件，包括影响球队、来源等级、影响因子、方向、强度和审核动作。
- 只有 `action=apply`、有明确球队、来源权重大于 0 的事件才进入模型。
- 社媒传闻和未证实消息不会直接改变概率。

第三层是产品解释层，负责把模型结果翻译成页面内容。

- 单场输出：胜 / 平 / 负概率、Top 3 最可能比分、比分矩阵、大小球和 BTTS 概率。
- 路径输出：如果主队胜、打平、客队胜，各自会怎样改变冠军概率和小组路径。
- 整届输出：冠军概率榜、今日概率变化榜、每轮晋级概率。
- 解释输出：五大盘面、新闻影响、来源权重、博主可讲选题。

### 2. 单场比分如何算

每支球队先有一个基础画像：

| 字段 | 含义 | 当前来源 |
|---|---|---|
| `elo` | 球队长期基础实力 | 自建 MVP 参数，可替换为授权评分 |
| `attack` | 进攻质量 | 自建公开代理指标 |
| `defense` | 防守质量 | 自建公开代理指标 |
| `goalkeeper` | 门将与防守边际 | 自建公开代理指标 |
| `squad` | 阵容健康和可用性 | 基础评分 + 事件修正 |
| `path` | 小组与淘汰赛路径难度 | 赛程结构 + 事件 / 上下文修正 |

模型会先根据双方 `elo` 差、进攻、防守、阵容和门将因素估算双方预期进球：

```text
主队预期进球 = 基础主队进球 × 实力差修正 × 主队进攻 × 客队防守暴露 × 主队阵容 × 客队门将拖拽
客队预期进球 = 基础客队进球 × 实力差修正 × 客队进攻 × 主队防守暴露 × 客队阵容 × 主队门将拖拽
```

然后用 Poisson 分布计算 `0-0` 到 `7-7` 的比分概率。比分概率加总后得到：

- 主胜概率：所有主队进球大于客队进球的比分概率总和。
- 平局概率：所有双方进球相同的比分概率总和。
- 客胜概率：所有客队进球大于主队进球的比分概率总和。
- 最可能比分：按比分概率从高到低取 Top 3。
- 比分矩阵：前端展示 `0-0` 到 `4-4` 的主要概率区域。
- 进球市场视角：从同一张比分分布里计算大 2.5、小 2.5、BTTS 是 / 否。

页面里的“模型公平概率”和“公平价格”只来自模型概率，其中公平价格按 `1 / 概率` 换算。它用于表达模型内部价格，不代表真实交易价格，也不构成投注建议。

### 3. 整届赛事如何模拟

整届赛事使用蒙特卡洛模拟，默认开发请求为 8,000 次，接口支持提高到 50,000 次用于正式更新。

每一次模拟会做这些事：

1. 保留所有已结束比赛的真实比分。
2. 对未开始比赛按单场比分分布随机采样比分。
3. 对进行中比赛保留当前比分，再采样剩余进球。
4. 计算 12 个小组积分、净胜球、进球数和胜场。
5. 每组前 2 名直接进入 32 强。
6. 12 个小组第 3 名里选出成绩最好的 8 支进入 32 强。
7. 按 2026 官方 32 强固定路径和 Annex C 小组第三组合表分配淘汰赛位置。
8. 淘汰赛用球队淘汰赛评分计算晋级方，并一路模拟到冠军。
9. 多次模拟后统计每队进入各阶段的比例，得到晋级概率和冠军概率。

淘汰赛评分使用：

```text
淘汰赛评分 = elo + 进攻修正 + 防守修正 + 阵容修正
```

两队相遇时，评分差会转成晋级概率。这样可以避免只靠单场比分模型造成淘汰赛路径过度随机，同时仍保留强弱差。

### 4. 权重因子

当前模型的权重因子分为五组。

| 因子组 | 进入模型的字段 | 影响位置 | 页面表达 |
|---|---|---|---|
| 基础实力 | `elo` | 单场预期进球、淘汰赛评分 | 实力盘 |
| 攻防质量 | `attack`、`defense` | 单场预期进球、比分分布、淘汰赛评分 | 状态盘 |
| 门将与边际 | `goalkeeper`、`defense` | 对手预期进球、低比分倾向 | 边际盘 |
| 阵容健康 | `squad` | 单场预期进球、淘汰赛评分 | 人员盘 |
| 路径难度 | `path`、小组排名、半区对手 | 晋级概率、冠军概率、路径传导 | 路径盘 |

前端展示的五大盘面是产品层映射：

```text
实力盘 = (elo - 1000) / 10
状态盘 = (attack + defense) / 2
路径盘 = path
人员盘 = squad
边际盘 = (goalkeeper + defense) / 2
```

### 5. 新闻和事件如何影响权重

新闻来源分为 S/A/B/C/D 五级：

| 来源等级 | 权重 | 处理方式 |
|---|---:|---|
| S | 1.0 | 官方赛程、官方伤病、官方纪律公告、官方首发，可以直接入模 |
| A | 0.7 | 权威媒体和专业数据源，可以入模，重大事件建议多源确认 |
| B | 0.4 | 可靠跟队记者或本地权威记者，谨慎小幅入模 |
| C | 0.2 | 普通媒体、自媒体、低确定性线索，默认待审核；多源确认后才可低权重入模 |
| D | 0.0 | 匿名爆料、未证实社媒传闻，不入模型 |

事件会被映射到五类模型因子：

- `attack`：前锋、边锋、进攻核心、创造机会相关信息。
- `defense`：中卫、后卫、防线、防守结构相关信息。
- `goalkeeper`：门将、扑救、出球稳定性相关信息。
- `path`：赛程、旅程、天气、高温、恢复时间相关信息。
- `squad`：伤病、停赛、首发、阵容可用性和队内稳定性。

事件修正公式：

```text
事件修正 = 影响方向 × 影响强度 × 来源权重 × 100
```

其中影响方向为 `+1`、`0` 或 `-1`。当前强度规则是：

- 停赛、缺席、受伤：`0.055`
- 主力、核心、连续两天训练异常：`0.035`
- 高温、天气、恢复：`0.020`
- 普通动态：`0.015`

示例：一条 B 级来源确认“巴西边路主力连续两天单独训练”，如果被归入 `attack` 且方向为 `-1`，则会产生：

```text
-1 × 0.035 × 0.4 × 100 = -1.4
```

也就是巴西进攻因子小幅下调约 1.4 分。这个修正会先影响单场比分分布，再传导到整届模拟。

### 6. 高阶数据代理和可替换数据源

当前版本没有直接接入未授权的 Opta、StatsBomb、Wyscout 等商业事件数据。为了让模型先跑起来，系统内置了一组自建公开代理指标：

- 面对强队胜率代理。
- 面对强队场均积分代理。
- 非点球 xG 代理。
- 防守 xGA 代理。

这些代理指标会被转成小幅 `attack`、`defense` 和 `path` 修正。后续如果客户有授权数据源，可以直接替换这些代理指标，不需要重写整条模型链路。

### 7. 赛前上下文如何进入模型

除了新闻事件，当前比赛上下文也会进入模型，主要影响路径盘和人员盘：

- 如果一队上一场到本场休息少于 4 天，会小幅下调。
- 如果两场比赛之间发生城市转场，会小幅下调。
- 这些影响会通过 `fixtureContextImpacts` 暴露给前端，用于解释“为什么某队路径盘或人员盘变化”。

### 8. 每日变化榜如何生成

日更流程会生成预测快照。今日概率变化榜不是简单写死文案，而是比较新旧快照：

- 冠军概率相比上一快照上调或下调多少。
- 每轮晋级概率有没有明显变化。
- 已结束赛果数量是否改变。
- 可入模型事件数量是否改变。
- 五大盘面是否出现新的正负修正。

因此页面可以解释“谁涨了 / 谁跌了”，也能解释“为什么涨跌”。

### 9. 当前模型限制

- 球队评分仍是自建 MVP 参数，不代表授权商业评级。
- 线上 Cloudflare Pages 当前是静态前端验收版，生产 API 尚未部署。
- 市场价格源尚未接入，页面只展示模型公平概率。
- 新闻自动抓取和赛果自动更新已有接口和脚本，但正式运营还需要稳定数据源与部署环境。
- 模型输出是概率分析，不是确定结论，也不是投注建议。

## 客户验收重点

客户打开线上地址后，建议重点看这几块：

1. 首页工作台是否能快速解释“今天该看哪场、为什么重要”。
2. 单场预测是否包含胜平负、最可能比分、比分矩阵、进球市场视角。
3. 路径传导是否能说明一场比赛如何影响小组排名、晋级路径和冠军概率。
4. 冠军概率榜和今日变化榜是否适合专业博主做每日内容。
5. 新闻影响和来源权重是否能解释概率变化原因。
6. 付费锁定区是否符合“赛前预测包 / 全周期查看”的商业形态。

## 给后续 Agent 的接手说明

先读这些文件：

- `AGENTS.md`：项目规则、模型边界、开发日志。
- `README.md`：客户和 Agent 的入口说明。

- `src/App.tsx`：主要前端页面和交互。
- `src/styles.css`：页面样式、响应式布局、移动端适配。
- `backend/model.py`：单场比分概率和整届模拟核心。
- `backend/main.py`：FastAPI 接口入口。
- `backend/data_files/*.json`：球队、赛程、事件、来源权重和当前比赛配置。
- `scripts/`：数据导入、日更、新闻导入、快照生成脚本。
- `tests/`：后端模型、接口、前端契约和部署检查。

```bash

npm run dev
```

```bash
npm run build

npm run validate:data
```

日更和快照：

```bash
npm run daily:update

```

## 合规边界

- 页面输出使用“概率”“模型公平概率”“市场价格源待接入”等表达。
- 不写下注建议，不承诺预测命中，不使用未授权商业数据。
- 官方结果、赛程、积分、红黄牌、首发可以进入模型。
- 伤病、训练、发布会和媒体报道必须区分来源等级。
- 社媒传闻原则上只做风险提醒，不直接大幅改变模型。
