## 一、项目目标

本项目要做的不是一篇静态的“世界杯夺冠热门预测”文章，而是一个可以持续更新的世界杯动态预测产品。

产品核心目标：

1. 每天根据真实比赛进展、赛程路径、球队状态和权威新闻，更新 2026 世界杯各队的夺冠概率、晋级概率和风险说明。
2. 让用户在移动端快速看懂：谁更可能夺冠，谁被高估，谁路径变难，谁因为伤病、停赛、新闻或赛程因素被下调。
3. 用稳定的底层概率模型承接专业性，用清晰的五大盘面承接用户理解，用新闻抓取模块承接每日动态。
4. 最终形态可以支持付费查看，例如单次查看、赛事全周期查看、重点场次预测包等。

本项目的判断边界：

- 这是概率分析产品，不是投注建议产品。
- 不承诺预测一定命中。
- 不使用未授权商业数据。
- 不暗示掌握内幕消息。
- 新闻抓取只基于公开来源，并且必须区分来源权威等级。

## 二、产品形态

### 1. 响应式 Web App，移动端优先

客户明确希望页面既能在电脑浏览器访问，也要方便手机浏览器访问。前端设计不能只做桌面端再缩小适配，而要从一开始按响应式 Web App 设计：桌面端展示完整数据视野，移动端提供单列滚动、触控优先、核心信息不丢失的浏览体验。

核心页面应包括：

- 单场比赛预测页。
- 今日夺冠概率榜。
- 今日概率变化榜。
- 球队详情页。
- 单队五大盘面分析。
- 新闻影响摘要。
- 小组和淘汰赛路径图。
- 付费解锁入口。

移动端必须保留的核心模块：

- 单场胜平负 / 晋级概率。
- 最可能比分和比分分布。
- 冠军概率榜。
- 五大盘面。
- 路径模拟摘要。
- 新闻影响和来源权重。
- 付费入口。
- 双击/双击触控后移动模块的提示。

视觉方向：

- 世界杯专题感。
- 绿茵场、赛程、队徽、概率热力图等体育视觉元素。
- 可参考腾讯体育、懂球帝、Opta Analyst、Football Meets Data 等产品表达，但不能复制页面设计和数据结果。

### 2. 用户看到的核心表达

用户不应该直接看到复杂公式，而应该看到能理解的解释：

```text
法国夺冠概率：13.4% → 12.1%

变化原因：
1. 核心前锋训练缺席，人员盘下调。
2. 小组第一路径仍较优，路径盘保持稳定。
3. 最近两场非点球 xG 下降，状态盘轻微下调。
```

页面要把复杂模型翻译成五大盘面：

| 盘面 | 用户理解 | 背后模型 |
|---|---|---|
| 实力盘 | 这队本来强不强 | Elo、SPI、Opta Power Rating、长期攻防强度 |
| 状态盘 | 这队最近好不好 | 最近比赛 xG、射门质量、防守质量、核心球员状态 |
| 路径盘 | 接下来这条路难不难 | 小组排名、潜在对手、半区强度、旅行消耗 |
| 人员盘 | 关键球员能不能上 | 伤病、停赛、负荷、替补可替代性 |
| 边际盘 | 细节有没有隐藏优势 | 定位球、门将、点球、天气、旅程、士气、裁判尺度 |

### 3. 用户到底在预测什么

本产品的预测对象不是单一的“谁会夺冠”，也不是单纯的“下一场谁赢”，而是：

```text
未开始比赛的单场结果走向
→ 影响小组排名和淘汰赛路径
→ 重算各队晋级与夺冠概率
→ 由 AI 生成可读的预测分析和变化原因
```

用户打开页面后，应该能获得三层预测：

| 层级 | 用户问题 | 页面输出 |
|---|---|---|
| 单场预测 | A 队对 B 队谁更可能赢 | 胜 / 平 / 负概率，淘汰赛展示双方晋级概率 |
| 比分预测 | 这场最可能踢成几比几 | 最可能比分 Top 3，比分概率分布 |
| 整届传导 | 这场结果会怎样改变世界杯走势 | 小组排名、路径难度、晋级概率、夺冠概率变化 |

比赛状态处理：

- 未开始比赛是核心预测对象。
- 进行中比赛如果没有可靠实时数据，第一版不强依赖实时修正。
- 已结束比赛不再预测，锁定真实赛果，作为后续概率和路径模拟的动态权重因子。

表达边界：

- 不把“比分概率”写成“赔率”。
- 不展示或承诺投注建议。
- 如果后续接入市场赔率，只作为市场热度或舆论偏差，不作为产品主预测。

## 三、模型总结构

本项目采用三层模型结构：

```text
底层概率模型
→ 中间事件与新闻处理层
→ 上层产品解释层
```

### 1. 底层：稳定概率模型

底层负责给出稳定、可复算的概率，不直接被新闻情绪带动。

建议底层路径：

```text
Elo / 实力评分
→ Poisson 或 Dixon-Coles 单场比分模型
→ 蒙特卡洛模拟整届世界杯
→ 输出夺冠、进决赛、进四强、进八强、出线概率
```

底层模型包含四个部分：

| 模块 | 作用 | 说明 |
|---|---|---|
| 球队强度评分 | 判断球队本身强弱 | 可参考 Elo、Opta Power Rating、SPI、FIFA 排名、最近比赛表现 |
| 单场进球模型 | 判断两队交手胜平负概率 | 可用 Poisson / Dixon-Coles，把强弱差转为比分概率 |
| 赛程路径模型 | 判断球队会遇到谁 | 根据小组排名、32 强路径、半区强度、潜在对手动态更新 |
| 蒙特卡洛模拟 | 反复模拟整届赛事 | 建议正式更新时至少模拟 50,000 次 |

底层输出：

- 小组出线概率。
- 小组第一、第二、第三概率。
- 进入 32 强、16 强、8 强、4 强、决赛、夺冠概率。
- 每场比赛胜平负概率。
- 每队路径难度。
- 每队概率变化幅度。

底层原则：

- 比赛已经结束后，必须锁定真实赛果。
- 已淘汰球队概率必须归零。
- 小组第三出线机制必须正确处理。
- 不把社媒传闻直接喂进底层公式。
- 赔率只作为市场预期参考，不等于真实概率。

### 2. 中间：事件与新闻处理层

中间层负责把比赛结果、伤病、停赛、训练、发布会、天气、旅程、队内动态等信息，转成结构化事件。

每条事件至少要包含：

| 字段 | 说明 | 示例 |
|---|---|---|
| 事件类型 | 信息类别 | 伤病、停赛、战术调整、士气、天气、赛果 |
| 影响对象 | 影响谁 | 法国队、姆巴佩、后防线、中场轮换 |
| 影响方向 | 利好、利空、中性、不确定 | 核心球员复出是利好 |
| 影响强度 | 轻微、中等、严重 | 核心前锋缺席淘汰赛是严重 |
| 来源等级 | 信息可靠性 | 官方、权威媒体、跟队记者、社媒 |
| 是否确认 | 确认状态 | 已确认、多源交叉、单源传闻、未证实 |
| 作用范围 | 是否进模型 | 直接修正、轻微修正、只做备注、等待确认 |

中间层处理三类信息：

1. 硬信息：比赛结果、积分、净胜球、红黄牌停赛、官方伤病公告。
2. 半硬信息：权威媒体伤病报道、训练缺席、主教练发布会、跟队记者消息。
3. 软信息：士气、队内气氛、更衣室传闻、社媒爆料、球迷观察。

处理原则：

- 硬信息可以直接修正概率。
- 半硬信息可以小幅修正，并等待后续确认。
- 软信息原则上只做风险提醒，不直接大幅改变概率。
- 多源交叉验证后，软信息可以升级为半硬信息。

### 3. 上层：产品解释层

上层负责把概率和事件翻译成用户能看懂的内容。

上层输出包括：

- 今日最大上调球队。
- 今日最大下调球队。
- 每队夺冠概率变化。
- 每队五大盘面评分。
- 今日关键新闻影响。
- 路径变难或变容易的原因。
- 关键球员风险提醒。
- 用户可读的模型解释。

上层不展示复杂公式，但要保留专业感。

## 四、因子重要程度

### 第一档：最高优先级，直接决定基础概率

#### 1. 球队基础实力

作用：判断球队是否具备夺冠硬实力。

可用指标：

- Elo。
- Opta Power Rating。
- SPI。
- FIFA 排名。
- 最近三年正式比赛表现。
- 对强队战绩。
- 预选赛和大赛表现。

开发处理：

- 第一版可以先建立球队基础评分表。
- 后续接入可更新数据源。
- 基础评分不应每天剧烈波动。

#### 2. 攻防质量

作用：判断球队能否稳定创造机会，同时限制对手机会。

可用指标：

- 非点球 xG / 90。
- 非点球 xGA / 90。
- xG 差值。
- xG per shot。
- 禁区内射门。
- 被对手进入禁区次数。
- 反击 xG。
- 定位球 xG。

开发处理：

- 如果第一版拿不到授权 xG 数据，可以先用公开进球、失球、射门、射正、近期战绩做替代。
- 后续有预算再接入正式 xG 或事件数据源。

#### 3. 晋级路径

作用：判断一支队实际要走的路有多难。

可用指标：

- 小组第一、第二、第三概率。
- 32 强潜在对手。
- 16 强潜在对手。
- 8 强潜在对手。
- 半区平均强度。
- 是否提前遇到顶级强队。
- 小组第三出线对应分支。

开发处理：

- 路径模型必须从第一版就做。
- 蒙特卡洛模拟要输出每轮概率。
- 每场比赛结束后路径必须重算。

### 第二档：高优先级，影响短期概率和淘汰赛上限

#### 4. 阵容健康与可用性

作用：判断关键球员能否连续出场。

可用指标：

- 官方伤病。
- 训练缺席。
- 过去 60 天出场分钟。
- 赛季累计负荷。
- 红黄牌停赛。
- 替补能否替代同一功能。

开发处理：

- 第一版可以手动录入关键伤病和停赛。
- 新闻抓取模块上线后，自动识别伤病和停赛事件。

#### 5. 主教练水平与临场调整

作用：判断球队是否能在淘汰赛高压下正确应对。

可用指标：

- 教练任期长度。
- 大赛淘汰赛经验。
- 换人后球队表现。
- 落后时追平能力。
- 领先后保护优势能力。
- 是否有 Plan B。

开发处理：

- 第一版不强行公式化。
- 先作为人工评分和备注因子。
- 后续可以根据历史比赛数据拆成量化项。

#### 6. 关键人物影响

作用：判断球星或核心节点对球队系统的影响。

可用指标：

- 关键球员 shot share。
- chance creation share。
- progressive carries。
- xA。
- 传球网络中心度。
- 核心缺阵时球队表现。
- 点球手稳定性。

开发处理：

- 第一版先维护每队 2-3 名关键人物。
- 新闻和伤病事件优先匹配关键人物。
- 关键人物事件的权重要高于普通替补事件。

### 第三档：中优先级，影响强队互打和淘汰赛细节

#### 7. 战术适配与风格克制

作用：判断不同风格对手之间是否存在克制关系。

可用指标：

- 控球压制能力。
- 高位压迫能力。
- 低位防守能力。
- 反击效率。
- 破密集防守能力。
- 是否怕高压。
- 是否怕反击。

开发处理：

- 第一版先用人工标签。
- 后续接入战术数据后再量化。
- 页面展示时可以转成“怕高压”“反击强”“定位球强”等标签。

#### 8. 凝聚力与阵容稳定

作用：判断球队内部是否能稳定执行计划。

可用指标：

- 主力阵容稳定度。
- 中卫搭档合作场次。
- 中场组合合作场次。
- 俱乐部队友块。
- 队长和核心是否清晰。
- 是否存在公开用人争议。

开发处理：

- 第一版作为人工备注。
- 新闻模块只在多源确认时影响概率。

#### 9. 定位球、门将与点球

作用：处理淘汰赛高杠杆细节。

可用指标：

- 定位球 xG。
- 定位球 xGA。
- 角球第一点成功率。
- 空中对抗能力。
- 门将扑救表现。
- 门将出球稳定性。
- 点球手质量。
- 点球大战经验。

开发处理：

- 第一版可以作为边际盘评分。
- 淘汰赛阶段提高该模块展示权重。

### 第四档：边际因素，强队接近时才放大

#### 10. 旅行、气候与恢复

作用：解释强队接近时的边际变化。

可用指标：

- 小组赛总飞行距离。
- 跨时区次数。
- 高温高湿场次。
- 高海拔场次。
- 比赛间隔天数。
- 训练基地到赛场距离。

开发处理：

- 第一版可以先基于赛程和城市手动计算。
- 后续接入城市坐标自动计算旅行距离。

#### 11. 市场与舆论偏差

作用：观察大众预期，不直接决定球队强弱。

可用指标：

- 博彩赔率变化。
- 预测市场价格。
- 媒体讨论量。
- 社交平台情绪。
- 热门队是否被过度追捧。

开发处理：

- 第一版可以暂不接赔率。
- 如接入赔率或预测市场，必须确认数据授权。
- 页面上只作为“市场热度”展示，不作为投注建议。

## 五、新闻抓取系统

### 1. 新闻模块定位

新闻模块不是用来替代模型，而是用来发现短期变化。

它负责：

1. 抓取公开新闻、官方公告、权威媒体、跟队记者消息。
2. 提取事件：伤病、停赛、训练、发布会、士气、天气、旅程、队内动态。
3. 判断来源权威等级。
4. 判断事件对哪支球队、哪名球员、哪个盘面有影响。
5. 给概率变化生成解释。

### 2. 新闻来源权威等级

| 等级 | 来源 | 是否进模型 | 处理方式 |
|---|---|---|---|
| S 级 | FIFA、世界杯官方、国家队官方、国家足协、官方首发、官方伤病、官方红黄牌 | 直接进入 | 权重最高，和低等级冲突时以 S 级为准 |
| A 级 | Opta、StatsBomb、The Athletic、BBC、ESPN、Sky Sports、L'Équipe、Reuters、AP、Guardian | 可以进入 | 重大事件建议多源交叉，伤病可中幅修正 |
| B 级 | 可靠跟队记者、本地权威记者、长期准确的实名记者 | 谨慎进入 | 单源轻微修正，多源后可提高权重 |
| C 级 | 普通媒体、自媒体、球迷社区、YouTube 分析、未知准确率 X 账号 | 不直接进入 | 只作为待确认线索 |
| D 级 | 匿名爆料、群聊截图、无出处搬运、“内部消息” | 不进入 | 默认忽略或标注未证实 |

### 3. 新闻事件权重

| 事件类型 | 影响强度 | 是否进入概率 | 示例 |
|---|---|---|---|
| 比赛结果 | 极高 | 直接进入 | 小组积分、净胜球、路径变化 |
| 红黄牌停赛 | 高 | 直接进入 | 主力中卫累计黄牌停赛 |
| 官方伤病公告 | 高 | 直接进入 | 核心前锋确认缺席 |
| 官方首发名单 | 高 | 进入单场模型 | 比赛前一小时更新胜率 |
| 权威伤病报道 | 中高 | 小幅进入 | 多家媒体称核心球员伤情未明 |
| 训练缺席 | 中 | 轻微修正或备注 | 连续两天缺席训练 |
| 主教练发布会 | 中 | 视内容进入 | 暗示轮换或确认伤情 |
| 战术调整报道 | 中 | 备注或小幅进入 | 权威分析称改打三中卫 |
| 更衣室矛盾 | 中低 | 原则上备注 | 多源确认后轻微影响凝聚力 |
| 士气变化 | 中低 | 原则上备注 | 队长发言、赛后情绪 |
| 天气与场地 | 低到中 | 小幅进入 | 高温高湿、高海拔、草皮问题 |
| 旅程与恢复 | 低到中 | 小幅进入 | 连续跨时区远距离飞行 |
| 社媒爆料 | 低 | 不直接进入 | 未证实传闻 |

### 4. 新闻处理流程

每日流程：

1. 抓取官方赛程、比分、积分、红黄牌。
2. 锁定已结束比赛结果。
3. 更新小组排名和路径概率。
4. 抓取官方伤病、发布会、训练和首发信息。
5. 抓取权威媒体与跟队记者新闻。
6. 把新闻转成结构化事件。
7. 对事件做来源等级判断。
8. 判断事件是否进入模型。
9. 重新跑概率模拟。
10. 生成面向用户的变化原因。

赛前流程：

1. 比赛前 24 小时更新天气、训练、伤病、停赛。
2. 比赛前 1 小时读取官方首发。
3. 更新单场胜平负概率。
4. 更新整届赛事蒙特卡洛模拟。
5. 输出赛前风险提示。

赛后流程：

1. 锁定比分。
2. 更新积分和路径。
3. 记录红黄牌、伤病、争议事件。
4. 读取比赛过程数据。
5. 更新状态盘。
6. 输出赛后概率变化说明。

## 六、算法与许可证边界

### 1. 可采用的方法

| 方法 | 是否可用 | 说明 |
|---|---|---|
| 蒙特卡洛模拟 | 可以 | 公开数学方法，可自己实现 |
| Poisson 模型 | 可以 | 公开统计方法，可自己实现 |
| Dixon-Coles | 可以 | 1997 年公开论文模型，可按公式实现 |
| Elo | 可以 | 常见评分方法，可自己实现 |

### 2. 可参考的项目

| 项目 | License / 属性 | 建议 |
|---|---|---|
| Cup26 AI / world-cup-2026-prediction-model | MIT | 可参考结构，谨慎复用代码，不复制页面和结果 |
| penaltyblog | MIT | 可评估作为 Python 建模库 |
| regista | GPL-3.0 | 不建议直接引入闭源商用项目 |
| Football Meets Data | 商业产品 | 只参考表达和方法，不复制数据和模拟器 |

### 3. 数据授权原则

算法可以自己实现，不等于数据可以随便使用。

相对安全：

- 官方赛程。
- 官方比分。
- 官方积分。
- 官方红黄牌。
- 官方首发。
- 官方发布会摘要。

高风险：

- Opta 详细事件数据。
- StatsBomb 付费数据。
- 商业赔率 API。
- 付费媒体全文。
- Football Meets Data 的模拟结果。
- 第三方平台专有概率。

第一版建议：

- 使用公开赛程和比分。
- 自建球队基础评分。
- 手动或半自动录入伤病、停赛和新闻事件。
- 不直接使用未授权商业数据。

### 2026-06-15：高阶球队强度分层与专业缺口补全

客户反馈：

- 现有 `elo / attack / defense / goalkeeper / path / squad` 六个核心变量适合做前台盘面，但作为底层预测特征过于笼统。
- 不希望用泛泛数据或编造数据填充模型，而是要把更专业的 8 个球队强度衡量层和 9 个专业系统差距真正写成函数，能在有真实数据时影响预测。

本次原则：

- 不伪造 xG、球员、首发、市场、战术、回测或贝叶斯数据。
- 已有真实输入只使用本地球队档案、官方赛程 / 赛果、已审核新闻事件和显式接入的授权高阶指标。
- 缺失的专业数据必须明确返回 `missing_authorized_data` 或 `neutral`，调整值为 0，不允许用“代理数”冒充真实高阶数据。
- 新增函数必须进入 `build_match_prediction()` 的 `modelMeta`，后台和后续 API 调用方可以看到每层是否接入、是否影响模型。

#### 已落地的 8 个球队强度细化层

新增文件：

```text
backend/team_strength.py
```

统一入口：

```text
build_team_strength_profile(team_key, teams, fixtures, metric_rows)
build_all_team_strength_profiles(teams, fixtures, metric_rows)
aggregate_team_strength_adjustments(teams, fixtures, metric_rows)
```

8 层定义：

1. `baseStrength`：基础强度层。  
   输入：本地球队档案中的 `elo / attack / defense / goalkeeper / path / squad`，以及可选授权字段 `fifaRating / independentElo / opponentAdjustedRating / marketPrior`。  
   当前作用：本地球队档案作为基础底盘，不额外制造调整；未来如果接入授权 FIFA rating、独立 Elo 或市场先验，会小幅修正攻防和路径。

2. `recentForm`：近期战绩层。  
   输入：当前 `fixtures.json` 中已完赛且带比分的官方赛果。  
   当前作用：根据已完赛比赛计算 `pointsPerMatch / goalsForPerMatch / goalsAgainstPerMatch / goalDifferencePerMatch`，并小幅修正 `attack / defense / path`。没有已完赛比赛时返回 `neutral_no_finished_fixtures`。

3. `elitePerformance`：强队对抗层。  
   输入：授权字段 `top20XgDiff / top20ResultIndex / daysSinceTop20Win`。  
   当前作用：有真实强队样本时修正 `attack / defense / path`；没有该数据时不影响模型。

4. `squadContinuity`：阵容相似性层。  
   输入：授权字段 `lineupContinuity / projectedXiStrength / injuryReplacementDropoff`。  
   当前作用：衡量预计首发与已验证强表现阵容的重合度、预计首发强度和伤停替代落差，修正 `squad / attack / defense`。

5. `attackQuality`：进攻质量层。  
   输入：授权事件数据字段 `npXgFor / shotQuality / boxEntries / transitionXg / setPieceXgFor`。  
   当前作用：有真实 xG / 射门质量时修正 `attack`；没有授权事件数据时缺失，不使用代理 xG。

6. `defenseQuality`：防守质量层。  
   输入：授权事件数据字段 `npXgAgainst / bigChancesAllowed / transitionXgAgainst / setPieceXgAgainst`。  
   当前作用：有真实防守过程数据时修正 `defense`；没有授权事件数据时缺失。

7. `goalkeeperQuality`：门将层。  
   输入：授权字段 `postShotXgMinusGoalsAllowed / claimCrossRate / sweeperActions / penaltySaveProfile`。  
   当前作用：衡量扑救超预期、处理传中、出击覆盖和点球能力，修正 `goalkeeper / defense`。

8. `tacticalProfile`：战术匹配层。  
   输入：授权字段 `pressingIntensity / pressResistance / directness / setPieceMismatch / aerialAdvantage`。  
   当前作用：衡量压迫、抗压、定位球和空中优势，修正 `attack / defense / path`。

#### 已落地的 9 个专业系统差距覆盖函数

统一入口：

```text
professional_gap_coverage(metric_rows)
```

9 个缺口：

1. `eventXgData`：真实 xG / xGA / 射门质量。  
   当前状态：如果没有授权 `attackQuality / defenseQuality` 字段，状态为 `missing`，模型影响为 `neutral`。

2. `playerModel`：球员级模型。  
   当前状态：依赖预计首发、球员评分、伤停替代落差；未接入时不影响模型。

3. `lineupPrediction`：首发预测。  
   当前状态：依赖 `lineupContinuity`；未接入官方首发或可信首发预测前不影响模型。

4. `eliteSampleAdjustment`：强队样本修正。  
   当前状态：依赖 Top 级对手结果和过程质量字段；有数据时进入 `elitePerformance`。

5. `marketCalibration`：授权市场校准。  
   当前状态：只接受显式授权 `marketPrior`；未接入时不使用市场赔率或非授权抓取。

6. `backtesting`：系统性回测。  
   当前状态：预留 `__meta__.backtesting`；未接入历史预测和真实赛果评估前只显示缺口，不调整概率。

7. `probabilityCalibration`：概率校准。  
   当前状态：预留 `__meta__.probabilityCalibration`；未接入 Brier score、log loss、校准曲线前不做温度缩放。

8. `bayesianUpdating`：动态贝叶斯更新。  
   当前状态：预留 `__meta__.bayesianUpdating`；未接入赛前 / 首发 / 实时事件分布前不改变现有确定性参数。

9. `tacticalMatchup`：战术 matchup 模型。  
   当前状态：依赖 `tacticalProfile`；没有真实压迫、抗压、阵型、定位球和空中优势数据时不影响模型。

## 十、当前交接摘要

一句话定义：

> 这是一个以未开赛比赛预测为入口、以整届世界杯概率变化为结果的动态预测系统。用户既能看单场谁更可能赢，也能看这场比赛会怎样改变各队晋级、路径和夺冠概率。

当前最重要的开发优先级：

1. 先把应用侧支付适配、等待页和三种本地模拟测试维持为可交付状态。
2. 客户拿到应用后自行配置微信 JSAPI、微信 Native 和支付宝创建订单 / 查单 / 鉴权 / 回调验签信息。
3. 客户字段如不一致，再按真实返回 JSON 做小范围字段兼容。
4. 补用户 / 设备 / 订单绑定，避免订单只能停留在本地意图层。
5. 把本地 JSON 订单迁移为正式数据库订单表和访问记录。
6. 持续替换自建球队评分和高阶代理指标，接入授权 xG / xGA / 事件数据。
7. 如找到 FIFA 官方红黄牌 / 完整首发专项接口，再把后台补录项升级为自动接入。

当前最重要的风险：

- 数据授权风险。
- 新闻传闻误导模型。
- 客户把产品理解成投注建议。
- 底层模型过复杂导致第一版无法快速交付。
- 页面只炫不清楚，用户看不懂概率为什么变化。
- 把“比分概率”误写成“赔率”，导致产品被误解为投注工具。

当前建议：

- 第一版先做“看起来专业、逻辑站得住、可每日更新”的 MVP。
- 不追求一次性把所有因子完全量化。
- 进行中比赛已支持手动或外部源写入比分，但不承诺强实时，除非后续接可靠实时数据源。
- 优先做未开赛单场预测和整届概率传导。
- 先把伤病、教练、核心球员、更衣室、天气旅程作为概率变化原因和人工修正项。
- 等真实数据和客户预算明确后，再逐步增强自动化与模型精度。
